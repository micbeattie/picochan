20// Main API for dev implementation, slightly higher level than
21// devib ones. They return negative error values on error
22// (e.g. -EINVAL). They do various parameter checks and return
23// errors instead of asserting like the low-level API does. Those
24// with cbindex_opt arguments leave the devib cbindex field alone
25// if called with a negative value, otherwise they validate it
26// as a callback cbindex and set the field or return a negative
27// error value, as appropriate. For sends (of data or zeroes), the
28// length sent is validated to be under the CSS-advertised window
29// (devib->size) and capped at that if not, with the actual count
30// returned. Many functions are variants of the full generic ones
31// that simply specialise the callback and flags fields.
32// Values between 1 and 255 are typically used to fit into the ASC
33// byte of a pch_dev_sense_t with sense code
34// PCH_DEV_SENSE_COMMAND_REJECT. ECANCEL is associated with sense
35// code PCH_DEV_SENSE_CANCEL.

207// dev API convenience functions with some fixed arguments:
208// * Omitting _then avoids setting devib callback by hardcoding -1
209// as the cbindex_opt argument of the full _then function.
210// * For send and send_zeroes family, the flags argument is set to
211// * PROTO_CHOP_FLAG_END for the _final variant,
212// * PROTO_CHOP_FLAG_RESPONSE_REQUIRED for the _respond variant
213// * 0 for the _norespond variant
214// * For pch_dev_update_status_ok family, call the corresponding
215// pch_dev_update_status_ function with DeviceEnd|ChannelEnd
216// * For pch_dev_update_status_error family, set devib->sense to the
